The most common failure mode isn’t a bad idea — it’s an idea that’s too big.

I’ve started “properly building” things and, a week later, realized I was designing architecture for a product that didn’t exist yet.

Now I use a simple filter. If it doesn’t pass, it stays in notes.

## My filter

- **Can I demo it in 30 seconds**: input → output. If it needs a presentation to explain, it’s too early.
- **Can I ship an MVP in 1–2 evenings**: not perfect, but working.
- **Is there one clear goal**: either a useful tool or a specific skill learned (e.g. “learn X”).
- **Is there minimal state**: fewer accounts/roles/DB usually means higher odds of finishing.

## How I cut scope

Rule: **1 core feature + 1 nice‑to‑have**.

If there’s a third item, it’s a plan — not a side project.

## What I actually put in my portfolio

Not “the hardest” code, but code that is:

- **easy to read** (no hidden magic)
- **easy to run** (1–2 steps and it works)
- **easy to show** (even if it’s small)

And yes: **shipping beats polishing**.
